## 背景

做了几年业务之后，我越来越能感受到：  
同一个团队里同时维护 React 和 Vue 项目，并不可怕。真正麻烦的，是两边项目的目录结构、状态管理、接口请求、组件命名和提交流程都不一样。

这样的问题一多，切换项目的成本会越来越高，新同学上手也会慢很多。

## 我最先统一的不是技术，而是约定

我通常会优先统一下面几件事：

- 页面目录怎么拆
- 接口层怎么组织
- 公共组件怎么沉淀
- 提交规范和分支规范怎么约定
- 环境变量和打包配置怎么管理

这些东西一旦统一，React 和 Vue 之间的切换成本会明显降低。

## 组件层怎么保持一致

我的习惯是，不强行要求两个框架写法一样，但会尽量让“职责边界”保持一致。

比如一个筛选区组件，不管在 React 还是 Vue 里，我都会让它只处理：

- 参数展示
- 用户交互
- 事件抛出

而不会在组件内部直接塞很多业务请求逻辑。

```ts
type FilterPanelProps = {
  value: SearchParams;
  onChange: (next: SearchParams) => void;
  onSubmit: () => void;
};
```

这种约定的好处是，团队在讨论组件设计时，重点会放在职责是否清晰，而不是陷入框架细节。

## 接口层尽量做薄

我比较不喜欢把接口调用写得到处都是。  
如果项目稍微复杂一点，我会要求接口层单独抽离，并在页面层只保留调用和状态处理。

这样做的价值很直接：

- 更容易排查问题
- 更容易复用接口逻辑
- 页面代码会更干净

## 团队协作里最有价值的一件事

我后来发现，最有价值的不是“选 React 还是选 Vue”，而是能不能把团队协作中的重复决策减少掉。

如果每个项目都从目录、命名、请求封装重新开始，那再熟悉的技术栈也会变得低效。

## 总结

同时维护 React 和 Vue 项目并不难，难的是在不同项目里持续保持清晰、稳定、低成本协作。  
这也是我现在越来越重视工程约定和可维护性的原因。
