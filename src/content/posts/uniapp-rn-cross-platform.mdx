## 为什么会同时接触 uni-app 和 React Native

在真实业务里，技术选型往往不是理想化的。  
有的项目更看重交付速度和多端覆盖，有的项目更在意交互自由度和原生体验。

所以我这几年里，既做过 uni-app 项目，也做过 React Native 项目。

## 我怎么理解这两类方案

### uni-app

更适合这类场景：

- 业务节奏快
- 需要尽快覆盖多端
- 页面型业务较多
- 团队更熟悉前端写法

### React Native

更适合这类场景：

- 交互要求更高
- 页面表现更灵活
- 对原生体验更敏感
- 需要更细的性能和组件控制

## 真正难的不是写页面

很多时候，跨端开发最难的不是把页面搭出来，而是下面这些问题：

- 多端差异怎么收口
- 状态同步怎么设计
- 公共组件怎么抽
- 调试链路是否顺手
- 发布和回归成本能不能接受

如果这些问题前期没想清楚，后面越做越容易痛苦。

## 我现在更看重的几个点

### 先判断业务是否适合跨端

不是所有项目都适合跨端。  
如果业务高度依赖原生能力、交互复杂度很高，硬做跨端只会让后面维护更吃力。

### 把公共能力抽清楚

像请求层、用户信息、埋点、主题、权限这类能力，越早抽清楚，后面越省事。

### 不要为了“统一”牺牲体验

我不太赞成为了多端统一，把本来该贴近平台体验的交互做得很僵硬。  
跨端的目标应该是提高交付效率，而不是让所有端都变得平庸。

## 总结

uni-app 和 React Native 没有绝对高下，关键还是看项目目标。  
我自己现在更倾向于把跨端开发理解成一种工程策略，而不是单纯的技术选型。
